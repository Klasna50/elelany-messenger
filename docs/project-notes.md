Non-obvious backend decisions in `supabase/schema.sql`:
- RLS recursion avoided via SECURITY DEFINER helpers `is_conversation_member` / `is_conversation_creator` / `is_conversation_owner`.
- `conversations.created_by uuid default auth.uid()` exists so an inserted row passes the SELECT policy for `.insert().select().single()` returns (creator not yet a member at insert time).
- Hybrid seen model: `messages.seen_at` (direct-chat single checkmark) PLUS `message_reads` per-user rows (group read receipts). RPC `mark_conversation_seen` writes both.
- 3 RPCs the app calls: `get_unread_conversation_counts`, `mark_conversation_seen`, `get_message_seen_summaries`.
- Storage buckets `avatars` + `chat-uploads` (public). Realtime enabled on messages/reactions/message_reads/conversation_members/call_signals.

- v1.1.7 (2026-07-22) — "Accept on a contact request does nothing". MY BUG in `respond_to_contact_request`: I declared a PL/pgSQL variable named `conversation_id`, which is ALSO a column of `conversation_members`. PL/pgSQL substitutes its variables into the `ON CONFLICT (conversation_id, user_id)` inference clause, so that statement failed at RUNTIME (function creation still succeeded, which is why the RPC probe looked healthy). Only Accept reached it — Ignore returns earlier — so Ignore worked and Accept didn't. RULE: in any SECURITY DEFINER function, prefix every local with `v_` so it can never capture a column name, especially in ON CONFLICT.
  Second fix in the same function: `direct_key` was built with the DB's default collation while the app uses JS `[a,b].sort()` (code-unit order). Those disagree on hyphenated strings, and every UUID has hyphens, so one pair could yield two different keys and two chats. Now `string_agg(v, ':' order by v collate "C")`.
  Client: the error was console-only so the card silently reappeared and the button looked dead. Now `contactRequestError` renders under the banner, and accepting opens the new chat (`fetchConversations` had to return `listItems` — additive, existing callers ignore it).

Probing deployed RPCs unauthenticated via `curl $URL/rest/v1/rpc/<fn>` with the anon key returns `{"status":"unauthenticated"}` and is a cheap way to confirm a function EXISTS (but not that its body runs).

- v1.1.9 (2026-07-22) — group history now starts at JOIN TIME (user's explicit choice: a new member must NOT see earlier messages). Apply with `supabase/group-history-from-join.sql`; also folded into schema.sql.
  KEY LESSON: changing `messages_select` was NOT enough. Three RPCs are SECURITY DEFINER and therefore bypass RLS entirely — all four had to move together, or the feature leaks/breaks:
  1. New helper `conversation_member_since(conv_id, uid)` (SECURITY DEFINER, stable) returns `conversation_members.created_at`, NULL for non-members.
  2. `messages_select` = `created_at >= conversation_member_since(conversation_id, auth.uid())`. NULL fails the comparison, so this REPLACES the membership check rather than sitting beside it.
  3. `get_unread_conversation_counts` += `m.created_at >= cm.created_at`, else a new member gets an unread badge for messages they cannot open.
  4. `mark_conversation_seen` += join-time filter, else senders are told a new member read pre-join messages.
  5. `get_message_seen_summaries`: `total_other_members` was ONE scalar for the whole conversation, so adding a member raised the target on every old message and they could never read as fully seen. Now a correlated per-message count over `cm.created_at <= m.created_at` (note: `group by m.id, m.created_at`).
  Existing members keep everything, because their membership row predates the messages. Someone removed then re-added starts fresh — intended.
  Client: `ChatListItem.joinedAt` (from `conversation_members.created_at`, added to the fetchConversations select) + `activeChatItem` / `joinedAfterConversationStarted` memos drive a "You joined on … — messages sent before that aren't shown" notice, gated on joined-started > 60s so founding members never see it.
  The migration file ends with a read-only report listing anyone who would lose history.
